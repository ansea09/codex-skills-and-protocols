# Current OpenAI-Facing Process

## Process Summary

From the OpenAI-facing viewpoint, interaction is organized as messages/messages that are assembled into model input/model input. At the conversation-description layer, the basic unit of communication is the message/message: a conversation/conversation is represented as a list of messages/messages, and each message has content/content plus source-like metadata.

In the Responses API, there is also a lower technical layer: `input` can be a string/string or a structured list of input items/messages. A product such as ChatGPT or Codex may also maintain conversation state/conversation state that is not identical to the full visible chat.

## Current Flow

1. The user/user writes a message/message through a product interface/interface such as ChatGPT or Codex, or through an application/application built on the OpenAI API.
2. The product/product or application/application builds a model request/model request.
3. That request may include the user input/user input, developer instructions/developer instructions, prior conversation history/history, available tools/tools, files/files, images/images, and other content/content.
4. OpenAI platform/platform may add system instructions/system instructions and product/runtime context.
5. The model/model receives a structured input/input containing messages/messages or other input items/input items.
6. The model/model applies the instruction hierarchy: root and system constraints outrank developer instructions, which outrank user instructions.
7. The assistant/assistant produces an assistant message/assistant message. This may be a final user-visible answer, or it may include a tool call/tool call.
8. If a tool/tool is called, the tool performs work outside the model and returns a tool output/tool output.
9. The tool output/tool output is added back to the interaction as tool-owned content/tool-owned content, and the model may be invoked again.
10. The assistant/assistant returns a response/response to the user/user through the same product or application boundary.

## Message Ownership And Authority

OpenAI documentation uses the field name `role`. In this model, the protocol-local interpretation is message ownership/source:

- `system`: message owned by the OpenAI/platform system layer;
- `developer`: message owned by the application developer/developer layer;
- `user`: message owned by the user/user or used as user-facing input/data;
- `assistant`: message generated by the assistant/assistant;
- `tool`: message generated by a tool/tool or programmatic execution.

Ownership/source is related to, but not identical with, authority/authority. System-level instructions have greater authority than developer instructions. Developer instructions have greater authority than user instructions. Assistant and tool messages are generally history, output, or data; they do not become higher-authority instructions unless a higher-authority instruction explicitly grants that status.

## System Message Vs Assistant Message

A system message/system message belongs to the OpenAI/platform system layer. It sets rules, safety boundaries, runtime constraints, product conditions, or tool availability. It is not the assistant replying to the user.

An assistant message/assistant message belongs to the assistant/model participant. It is the model's generated response, intermediate answer, or tool-call message. It may become part of conversation history, but it does not have the same instruction authority as a system or developer message.

The system role/system role is a rule-setting and boundary-setting role. The assistant role/assistant role is an answering and acting role enacted inside those boundaries.

## Conversation And Conversation State

A conversation/conversation is a sequence or persisted object that connects multiple turns/turns of interaction. In a ChatGPT or Codex UI, a conversation often appears as one chat/thread. However, the model-visible input on a given turn may be only a slice of that chat, a compacted summary, a reconstructed state, or a chain linked by response identifiers.

Conversation state/conversation state is saved information about earlier interaction turns that lets a later model call continue the same interaction rather than starting from empty context. It can be represented by explicitly supplied history/history, a `previous_response_id` chain, or a persistent conversation object/conversation object. It is an application/API state record, not human-like memory inside the model.

## Product Surface Note: ChatGPT And Codex

ChatGPT and Codex are product surfaces over OpenAI models and runtime services. ChatGPT is a general assistant-facing product. Codex is OpenAI's coding agent surface, where interaction may include workspace files, shell commands, browser/tool use, GitHub context, and code changes. Those product-specific capabilities affect available tools and side effects, but the message/input/authority distinctions above still apply.
